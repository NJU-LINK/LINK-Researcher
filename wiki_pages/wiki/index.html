Technical Wiki / Project-Centric Overview

Overview: 面向真实交付的多 Agent 研究生产线

LINK-Researcher 的核心不是“生成一段答案”，而是把复杂研究任务拆解为可执行、可观察、可收敛的流水线。 系统围绕 LangGraph 进行节点编排，覆盖任务理解、规划、检索、并行写作、后处理与校验，最终输出可交付结果。

Multi-Agent Collaboration

角色定义、规划、采集、格式化、校验等 Agent 协同执行

Parallel Processing

并行数据采集与并行章节生成，缩短长链路耗时

Tool-Integrated Retrieval

Web Search / Web Fetch / ArXiv / Image / MCP FileSystem

Config-Driven Runtime

FAST_MODE 与模型映射配置支持速度-质量权衡

从输入到交付：主流程骨架

START
  -> role_play
  -> planner
  -> (perception | page_replan)
  -> data_collection (parallel)
  -> init_design_guide
  -> init_format
  -> format (parallel)
  -> post_process
  -> validation
  -> zip_data
  -> END
为什么这条链路有效： 它把“长链路不确定性”拆成清晰阶段，并通过状态管理与并行机制提升稳定性和吞吐。

关键模块与代码落点

能力作用关键文件

运行取舍：速度、质量与稳定性

  • 速度：FAST_MODE 可减少搜索与推理开销，适合快速迭代。
  • 质量：保留 validation 与更充分检索，通常能提升结构一致性与证据完整性。
  • 稳定：通过显式节点边界 + 状态追踪，故障更容易定位与修复。
  • 扩展：新增工具、模型或格式时，可在现有图谱框架下增量演进。

1. 快速启动

主入口是 run_flow_fastapi.py。当前项目支持 html 与 md 两种图谱路径。

python run_flow_fastapi.py \
  --session_id "$(uuidgen)" \
  --prompt "分析近三年多模态大模型研究趋势" \
  --result_format "html" \
  --human_in_loop False
实现观察： 入口中会初始化 Redis、工作区、MCP 客户端状态检查、搜索服务探活，再启动图谱执行与消息队列处理。

2. 系统架构

编排层（Graph）

src/graph/base_graph.py 定义公共节点与路由，格式子类覆盖格式特定 Agent。

  • 通用节点：role_play / planner / data_collection / perception
  • 格式节点：init_format / format / post_process / validation
  • 收尾节点：zip_data

能力层（Tools）

组合 Web Search、Web Fetch、Image Search、ArXiv、FileSystem(MCP) 等工具。

  • src/tools/research/web_search.py
  • src/tools/research/web_fetch.py
  • src/tools/research/arxiv_search.py
  • src/tools/research/file_system.py

模型层（Models）

config/agent_config.py 中按 Agent 绑定模型配置，ModelManager 统一加载模型参数。

  • 默认：gpt-5.4
  • 视觉：qwen3-vl-plus
  • 回退：qwen3-235b-a22b-thinking-2507

3. 工作流拓扑

从 BaseGraph.build_graph() 可还原核心链路：

START
  -> role_play
  -> planner
  -> (perception | page_replan)
  -> data_collection (并行)
  -> init_design_guide
  -> init_format
  -> format (并行)
  -> post_process
  -> validation
  -> zip_data
  -> END
Phase A

任务语义澄清 + 角色定义

Phase B

计划生成 + 感知输入文件

Phase C

并行搜集证据 + 结构化写作

Phase D

后处理、校验与打包结果

4. 核心模块拆解

4.1 图谱路由层

src/entrypoint/graph_imports.py 根据 result_format 分流到 HTMLGraph / MDGraph。

这意味着同一套研究前置流程可复用，不同输出格式只需覆盖格式相关 Agent 与打包逻辑。

4.2 并行任务分发

send_to_data_collection() 与 send_to_format() 用 Send() 批量派发子任务，实现并行数据收集和并行章节写作。

4.3 可选角色与校验环节

ENABLE_ROLE_PLAY、ENABLE_VALIDATION 让流程可在质量与速度之间切换。

6. 运行时与状态管理

  • 任务状态：src/states/task_status.py 追踪阶段与中断信息。
  • 会话内存：src/memory/memory_manager.py 保存 plan、上下文与中间结果。
  • 消息队列：MessageQueueProcessor 负责事件流输出与结束通知。
  • MCP 连接：启动前检查健康，异常时自动重试并清理连接。

7. 扩展开发指南

新增格式输出

继承 BaseGraph，实现格式 Agent 与 get_zip_data_node()，再在 graph_imports.py 注册。

新增工具能力

在 src/tools/research/ 增加工具并注入到对应 Agent 工具列表。

新增模型策略

在 config/model_config.py 注册，随后在 config/agent_config.py 绑定到指定 Agent。

8. 常见问题

为什么开启 FAST_MODE 后效果变化明显？

因为 FAST_MODE 会联动关闭深度思考与验证，降低检索数量，并启用章节压缩。

为什么 markdown 与 html 路径行为不同？

两者前置研究节点共用，但格式 Agent、后处理与打包逻辑由各自 Graph 子类实现。
